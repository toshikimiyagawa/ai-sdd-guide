# 00. 概要 — なぜSDDか

## このガイドの目的
- 新規プロジェクトに submodule で取り込み、AIエージェントに開発ルールを守らせる。
- 人間が「どう開発すべきか」を体系的に理解できるようにする。

## スペック駆動開発 (SDD) とは
「先に仕様 (spec) を決め、それを契約として実装・検証する」開発スタイル。
コードを書く前に「何を・なぜ・完了条件は何か」を明文化することで、手戻りと、
AIエージェントの暴走（スコープ膨張・的外れな実装）を防ぐ。

## 中心思想：設計と実装の分離
- **設計（spec/plan/tasks/レビュー）= superpowers 必須（どのエージェントでも可）。** superpowers と subagent をフル活用して深く考える。
- **実装 = どのエージェントでも可。** 設計成果物 `specs/<feature>/` を唯一の契約として渡す。

この分離により、設計フェーズは superpowers が使えるエージェントが担当し、実装は Cursor / Copilot / Codex など
好きなツールで分担できる。そのため設計成果物には「設計の文脈を知らないエージェントでも
実装できる」品質が要求される。これは制約だが、仕様の曖昧さを早期に潰す良い圧力になる。

## 全体像
| 要素 | 対象 | 言語 | 役割 |
|---|---|---|---|
| `rules/` | agent | 英語 | 常時参照する強制ルール |
| `docs/` | 人間 | 日本語 | 方法論の解説 |
| `templates/` | 両方 | - | spec/plan/tasks 雛形 |
| `integration/` | 取り込み側 | - | ルート入口・hooks・CI・subagent定義 |
| `orchestration/` | 両方 | - | マルチエージェント・オーケストレーション |
| `catalog/` | 両方 | - | 機能単位カタログ |

## 守らせる仕組み（多層）
1. **Claude hooks** — 設計フェーズの逸脱をローカルで弾く（速い注意喚起・バイパス可）。
2. **CI** — 全エージェント共通の権威ある門番（spec有無・テスト・受入条件）。バイパス不可。
3. **sdd-reviewer subagent** — 実装が凍結specに適合するか独立に監査。
